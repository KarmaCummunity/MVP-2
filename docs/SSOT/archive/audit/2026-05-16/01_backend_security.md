# Audit 2026-05-16 — Backend security

## Summary

Overall posture is good. The 0070 hardening migration effectively closed every CRITICAL/HIGH item it claimed, but the subsequent 0072 RLS rewrite reintroduced one HIGH regression (lost the `removed_at IS NULL` guard from 0028's null-safe rewrite) and exposed a new PostgREST enumeration surface (`is_active_member`). 0083's `post_actor_identity` is structurally sound but the SELECT RLS leaks counterparty user_ids that the identity-exposure projection is meant to hide. Total NEW findings: 14 — 2 HIGH, 8 MEDIUM, 4 LOW. No CRITICAL.

## Re-verification of 2026-05-10 followup §1.* and §15.*

- §1.3 (`users_select_public` block-check noop for anon) — confirmed effectively closed: 0070 revoked anon RPC access to `is_blocked`. No regression in 0071–0084.
- §1.5 (Realtime publication on `users` exposes denormalized counters) — still open. No change since 2026-05-10. `0007_users_realtime_publication` is unchanged. New row from §15.7 (same issue).
- §1.6 (`is_post_visible_to()` redefined across 0002/0003/0005 without versioning) — still open. `0059_post_visibility_closed_public` redefines again without a `drop function if exists` cycle. Same pattern.
- §1.7 (Storage buckets world-readable) — accepted MVP trade-off (TD-11); not re-flagged.
- §1.8 (`find_or_create_support_chat()` rate-limit) — still open; 0074 added a NEW caller (`rpc_submit_support_issue`) that inherits the same gap.
- §1.9 (`false_reports_count` one-way) — still open.
- §1.10 (Migrations re-create without revoke-and-grant cycle) — still open; 0070 itself uses the safe revoke-then-grant pattern (good), but 0072 and 0083 do not.
- §15.7 (Realtime users-publication leaks counters) — still open. No mitigation since 2026-05-10.
- §15.9 (`messages.sender_id ON DELETE SET NULL` keeps body) — still 🟡 documented intent.
- §15.14 (`is_post_visible_to()` STABLE per-row evaluation) — still open. 0059 inherits the per-row cost.

## Hardening migration 0070 verification

- §1.1 anon SELECT on `follow_edges` — **confirmed closed**. `revoke select on public.follow_edges from anon;` at line 45. No subsequent migration re-grants.
- §1.2 / §15.2 PostgREST-enumerable predicate functions (`is_admin`, `is_blocked`, `is_following`, `has_blocked`, `active_posts_count_for_viewer`) — **confirmed closed** via the explicit triple-revoke (lines 60-69). However see **BSEC-2**: 0072 reintroduces a sibling (`is_active_member`) with the exact pattern that was just closed.
- §15.11 `inject_system_message` client-callable — **confirmed closed**. EXECUTE revoked from public/anon/authenticated. 0074's `rpc_submit_support_issue` legitimately calls it as a SECURITY DEFINER wrapper, which bypasses the missing grant via owner privileges.
- §15.3 support-thread race — **partially closed**. The public RPC `rpc_get_or_create_support_thread()` is now atomic (good). But the internal helper `find_or_create_support_chat(uuid)` (0033) still has the read-then-insert race and is the one called by `rpc_submit_support_issue` (0074) and several report triggers. See **BSEC-5**.
- §15.5 `posts.reopen_count` server-managed — **confirmed closed**. Column dropped from the authenticated UPDATE grant (line 153-158); `reopen_post_deleted_no_recipient` RPC backfills the missing flow. Companion FE call confirmed in `closureMethods.ts:93`.
- §15.1 `reports.resolved_at` / `resolved_by` server-managed — **confirmed closed**. Same pattern (lines 168-169).

## Notes / non-issues investigated

- Pre-existing SECURITY DEFINER functions missing `SET search_path`: only `public.set_updated_at` (0001:88), `public.follow_requests_on_status_change` (0003), `public.messages_on_status_change` (0004:213). Pre-existing tech debt, not new in 0070+. Worth a sweep but not a security hole because these functions are simple triggers operating on `NEW`.
- `cities` table RLS — 0082 added an explicit comment clarifying that "select using (true) + RLS on" is the intentional pattern (writes blocked by default). Confirmed not theatrical.
- `pg_cron` jobs (0066, 0076, 0077, 0079, 0080) — all use `do $mig$ ... $mig$` outer-tag pattern to avoid CLI tokenizer collisions; the inner cron bodies reference `app.settings.functions_url` and `app.settings.service_role_key` GUCs and abort with a warning when unset. Reviewed for shell-style injection — none possible (no string concatenation with user data).
- `dispatch-notification` Edge Function — service-role bearer required; uses the typed coalesce helpers; recipient suspension is honored (line 76); `DeviceNotRegistered` properly cleans up the row. No new issues.
- `reconcile-storage-orphans` and `strip-exif` — both correctly require the service-role bearer (literal `===` comparison after Bearer strip). EXIF-strip parser bounds-checks segment lengths before slicing.
- `delete-account` Edge Function — correctly uses USER client (Authorization passthrough) for the RPC so `auth.uid()` is populated, then service-role for storage and `auth.admin.deleteUser`. Recommended order: DB → storage → auth (preserves idempotency on retry).
- Storage policies for `post-images` UPDATE/DELETE — `WITH CHECK` is missing from the UPDATE policy but the standard Supabase pattern relies on USING also gating WITH CHECK semantically; the path-rename attack lands a user in a path whose first segment must still match their own `auth.uid()::text`, so practically safe.
- Realtime publication for `post_actor_identity` — table not added to `supabase_realtime` (no `alter publication` call in 0083). Reads happen on-demand via fetch. ✓
- 0078 backfill migration — read-only over `messages`, no privilege escalation surface.
- 0081 Hebrew street-number constraint — uses safe `%I` identifier quoting; no SQL injection.
