# 6.6 Audit Trail

[← back to Part VI index](./README.md)

---

## Purpose

The audit trail is the immutable record of who did what, when, and to whom — for security investigations, dispute resolution, and policy compliance.

The product rule that mandates an audit log is `R-MVP-Safety-3`. This document specifies the **schema**, **action taxonomy**, **retention**, and **access** for the trail.

---

## 6.6.1 Schema

The audit log lives in the `audit_events` table:

```
event_id          uuid (PK)
actor_id          uuid | null   (null when system-originated)
actor_kind        enum: user | system | super_admin
action            text          (one of the values in 6.6.2)
target_type       enum: user | post | chat | message | follow_request | follow_edge | block | report | recipient | feature_flag | none
target_id         uuid | null
metadata          jsonb         (action-specific payload)
ip_hash           text | null   (sha256 truncated of source IP, for abuse forensics)
session_id        uuid | null
created_at        timestamptz   (immutable, server-set)
```

### Schema invariants

- `audit_events` is **append-only** at the database layer (`UPDATE` and `DELETE` are revoked from all roles in production; only an explicit retention job may delete).
- The `event_id` is generated by the application layer and used for idempotency (a retried operation produces a single audit row).
- The `metadata` JSON has a per-action shape documented per row in 6.6.2.

## 6.6.2 Action taxonomy

The complete enum of audit actions in MVP. Adding a new action requires updating this list.

### Auth

- `auth_signup` — `metadata: { method }`.
- `auth_signin` — `metadata: { method, success: bool, failure_reason? }`.
- `auth_signout` — *no extras*.
- `auth_password_reset_requested` — *no extras*.
- `auth_password_reset_completed` — *no extras*.
- `auth_account_deleted_requested` — *no extras*. (initial action by user)
- `auth_account_purge_executed` — `metadata: { tenure_days }`. (system, after 30d)

### Profile

- `profile_updated` — `metadata: { fields: string[] }`.
- `profile_privacy_changed` — `metadata: { from, to, pending_request_count }`.
- `profile_avatar_imported_from_sso` — `metadata: { provider }`.

### Posts

- `post_created` — `metadata: { type, visibility, category }`.
- `post_edited` — `metadata: { fields_changed: string[], visibility_upgraded: bool }`.
- `post_deleted` — `metadata: { tenure_open_seconds }`.
- `post_visibility_changed` — `metadata: { from, to }`.
- `post_expired` — *system action*.
- `post_auto_removed` — `metadata: { trigger: 'three_reports' | 'admin' }`.
- `post_restored` — `metadata: { admin_id }`.

### Closure

- `post_closed_with_recipient` — `metadata: { recipient_id }`.
- `post_closed_no_recipient` — *no extras*.
- `post_reopened` — `metadata: { previous_status }`.
- `recipient_unmarked_self` — `metadata: { post_id, owner_id }`.

### Follow

- `follow_started` — `metadata: { followed_id, target_was_public: bool }`.
- `follow_request_created` — `metadata: { target_id }`.
- `follow_request_resolved` — `metadata: { result: 'accepted' | 'rejected' | 'cancelled' }`.
- `follower_removed` — `metadata: { follower_id }`.
- `unfollow` — `metadata: { followed_id }`.

### Chat

- `chat_message_sent` — `metadata: { chat_id, length_bucket }`.
- `chat_message_failed_delivery` — `metadata: { reason }`.

### Moderation

- `report_submitted` — `metadata: { target_type, target_id, reason }`.
- `report_resolved` — `metadata: { result: 'confirmed_violation' | 'dismissed_no_violation' }`.
- `block_user` — `metadata: { blocked_id, surface }`.
- `unblock_user` — `metadata: { blocked_id }`.
- `false_report_sanction_applied` — `metadata: { tier: 1|2|3, duration_days }`.
- `user_suspended` — `metadata: { reason, until_date }`.
- `user_unsuspended` — *no extras*.
- `user_banned` — `metadata: { reason }`.

### Admin / system

- `feature_flag_changed` — `metadata: { key, from, to }`.
- `super_admin_login` — *security-critical*.
- `audit_log_accessed` — `metadata: { target_user_id }`. (when the Super Admin reads another user's audit history)

## 6.6.3 Retention

- Audit events are retained for **24 months**.
- After 24 months, rows are **anonymized in place** rather than deleted: `actor_id` becomes `null`, `ip_hash` cleared, `metadata` keys carrying user IDs are nulled. The action and timestamp remain for aggregate trend analysis.
- This anonymization is performed by a quarterly maintenance job documented in [`05_background_jobs.md`](./05_background_jobs.md) (added in V1.5; tracked as `Q-006` in [`appendices/D_open_questions.md`](../appendices/D_open_questions.md) for MVP).

---

## 6.6.4 Access

- Read access to `audit_events` is restricted by RLS to:
   - The acting user (read their own actions only).
   - The Super Admin (read everything, with `audit_log_accessed` self-recording).
- Write access is restricted to a database role used only by Edge Functions and trigger-based emitters; clients never write directly.

---

## 6.6.5 Emission patterns

Audit emission is **synchronous within the transaction** of the audited operation. A failure to write the audit row aborts the operation. This is by design: a successful operation without an audit trail violates `R-MVP-Safety-3`.

The `application` layer exposes an `Audit` port (interface) consumed by every use case that performs an audited action. Implementation lives in `infrastructure-supabase`.

---

## 6.6.6 PII in audit metadata

- `metadata` may include **internal IDs** (`post_id`, `recipient_id`, etc.) but never raw PII (names, emails, phones, addresses, message bodies).
- IP addresses are stored only as truncated SHA-256 hashes for security forensics; raw IPs never appear in the audit log.

---

## 6.6.7 Tamper-evidence

- Database triggers enforce immutability (`TRUNCATE` is also revoked).
- A monthly job computes a Merkle hash of the audit log per month and stores it in a separate write-once-read-many bucket; tampering would change the hash.
- Tamper-evidence enforcement is operational; documented in `CODE_QUALITY.md`.
